- If Probe A returns a JSON object containing a formatted DAX string (not HTML, not 404) → use JSON endpoint in skill, document it as confirmed in api-notes.md

- If both probes fail → skip the API call in the `!` bash injection and go straight to Claude inline fallback; document "API unreachable as of [date]" in api-notes.md

Instructions block:

1. Initial response: "Paste your DAX measure below:"

2. Measure extraction: Same rule as /pbi:explain — extract measure name from before first `=`; if no `=`, use `[Measure]` placeholder with note.

3. Formatting logic — check the "Format API Status" section above:
   - If API_OK: Use the confirmed API endpoint (documented in api-notes.md) to format the pasted measure. Extract the formatted DAX string from the API response. Return it in a fenced ```dax code block.
   - If API_FAIL: Add exactly this one line at the top of the output: `_DAX Formatter API unavailable — formatted inline by Claude._` Then format the measure using Claude's SQLBI-style rules (below) and return it in a fenced ```dax code block.

4. Claude inline SQLBI formatting rules (used when API_FAIL or as fallback):
   - All DAX keywords in UPPERCASE: CALCULATE, SUMX, FILTER, ALL, VALUES, RELATED, DIVIDE, IF, BLANK, VAR, RETURN, etc.
   - Each CALCULATE argument on its own line, indented 4 spaces
   - Each VAR/RETURN on its own line
   - Opening parenthesis stays on same line as function name
   - Closing parenthesis on its own line, aligned with function name
   - Measure name and `=` on first line, then expression body indented
   - Example:
     ```dax
     Sales YTD =
     CALCULATE(
         [Revenue],
         DATESYTD('Date'[Date])
     )
     ```

5. Output structure:
   - (If API_FAIL): `_DAX Formatter API unavailable — formatted inline by Claude._`
   - Fenced ```dax code block with the formatted measure
   - Next steps line: `**Next steps:** /pbi:explain · /pbi:optimise · /pbi:comment · /pbi:error`
