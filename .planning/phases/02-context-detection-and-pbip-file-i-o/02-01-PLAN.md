File 2 — definition/tables/Sales.tmdl:
A TMDL table file for the Sales table. Must include:
- Exactly two measures: one with a triple-slash description already present, one without a description (to exercise both code paths)
- At least two columns (for pbi-load column extraction)
- Correct TMDL syntax: tab indentation (not spaces), measure keyword, triple-slash description on the line IMMEDIATELY above the measure keyword (no blank line between)

Use this content:

```tmdl
table Sales

	column Date
		dataType: dateTime
		formatString: General Date

	column Amount
		dataType: decimal
		formatString: #,##0.00

	/// Year-to-date revenue filtered to visible periods.
	measure 'Revenue YTD' =
			CALCULATE([Revenue], DATESYTD('Date'[Date]))
		formatString: #,##0
		displayFolder: "Time Intelligence"

	measure Revenue =
			SUM(Sales[Amount])
		formatString: #,##0
		displayFolder: "Base Measures"
```

Critical indentation rule: TMDL uses tabs. Table-level declarations (column, measure) are indented one tab. Measure expression content and properties are indented two tabs. Do NOT convert tabs to spaces.

File 3 — definition/relationships.tmdl:
A minimal relationships file. Used by pbi-load to extract the relationships summary.

```tmdl
relationship 'Sales_Date'
	fromTable: Sales
	fromColumn: Date
	toTable: Date
	toColumn: Date
	crossFilteringBehavior: bothDirections
```

Note: crossFilteringBehavior: bothDirections is intentionally included — it is a bidirectional relationship that pbi-audit (Phase 3) will flag. Having it in the fixture now makes Phase 3 fixture work easier.

File 2 — model.bim:
A minimal TMSL JSON file. Must include:
- One table (Sales) with two columns and two measures
- One measure with a description field already populated, one without
- The expression field for the multi-line measure must be an ARRAY of strings (not a single string) — this is the form Power BI Desktop uses for expressions with line breaks, and the skill must preserve this form on write-back
- The expression field for the simple measure must be a plain string — exercises both expression forms

Use this content:

```json
{
  "name": "SalesModel",
  "compatibilityLevel": 1550,
  "model": {
    "culture": "en-US",
    "tables": [
      {
        "name": "Sales",
        "columns": [
          {
            "name": "Date",
            "dataType": "dateTime"
          },
          {
            "name": "Amount",
            "dataType": "decimal"
          }
        ],
        "measures": [
          {
            "name": "Revenue YTD",
            "expression": [
              "CALCULATE(",
              "    [Revenue],",
              "    DATESYTD('Date'[Date])",
              ")"
            ],
            "description": "Year-to-date revenue.",
            "formatString": "#,##0",
            "displayFolder": "Time Intelligence"
          },
          {
            "name": "Revenue",
            "expression": "SUM(Sales[Amount])",
            "formatString": "#,##0",
            "displayFolder": "Base Measures"
          }
        ]
      }
    ],
    "relationships": [
      {
        "name": "Sales_Date",
        "fromTable": "Sales",
        "fromColumn": "Date",
        "toTable": "Date",
        "toColumn": "Date",
        "crossFilteringBehavior": "bothDirections"
      }
    ]
  }
}
```

Note: The "Revenue" measure intentionally has no "description" field — this exercises the add-description code path in pbi-comment file-mode. The "Revenue YTD" measure has description populated — exercises the update-description code path.
